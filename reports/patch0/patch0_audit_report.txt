CHAKANA STUDIO - PATCH 0 AUDIT
Date: 2026-03-12
Scope: main menu, map/nodes, combat HUD, rewards/pre-shop/shop, codex, hologram dialogue, transitions, intro/studio/credits
Method: targeted audit of active runtime screens, shared UI systems, dialogue/audio routing, and known QA wrappers. No full-repo scan.

Executive Summary
- The current runtime stack is concentrated under game/ui/screens plus shared systems in game/ui/system, game/ui/components, game/ui/layout, game/visual, and game/audio.
- The project no longer looks like a single old UI; it looks like a newer runtime layer with several fallback, alias, and legacy compatibility paths still active.
- Patch 0 should focus on system unification and readability, not new content production.
- The highest-risk problems are duplicated presentation paths, fallback-heavy asset/audio routing, and legacy labels/flows still visible in user-facing screens.

Severity Scale
- blocker: breaks flow consistency, causes conflicting systems, or risks wrong content/context presentation
- major: strong UX quality issue, duplicated implementation, or layout/system debt that spreads defects
- medium: visible inconsistency or maintainability issue with limited runtime damage
- polish: copy/style cleanup, visual refinement, or non-blocking consistency work

Domain Audit

1. Main Menu
Primary files
- game/ui/screens/menu.py
- game/ui/screens/intro.py
- game/ui/screens/studio_intro.py
- game/ui/system/brand.py
- game/ui/system/typography.py

Mapped issues
- major: menu.py mixes current premium title treatment with old generic menu button flow. The screen is stable, but it still behaves like a generic launcher rather than a fully authored front door for the current identity.
- medium: intro.py is a separate static narrative intro with a heavy opaque panel, while studio_intro.py is a more authored branded intro. This creates two different intro identities in the same stack.
- medium: menu button stack still exposes generic labels/flow ordering inherited from older app navigation instead of a fully intentional game-first presentation.
- polish: version line and title treatment are consistent technically, but the screen still relies on broad centered-column utility layout rather than a dedicated menu layout system.

Duplicated or old systems still active
- intro.py and studio_intro.py both act as top-of-funnel presentation layers.
- studio_intro.py still keeps explicit fallback modes, cache/generation logic, and static fallback rendering.

Legacy UI layers / labels / transitions
- intro.py panelized text presentation reads as older placeholder narrative UI compared with studio_intro.py.
- menu text keys still reflect a generic app shell: continue, back, settings, codex, exit.

Severity
- overall: major

2. Map / Nodes
Primary files
- game/ui/screens/map.py
- game/ui/components/topbar.py
- game/ui/system/layout.py
- game/ui/system/icons_atlas.py

Mapped issues
- blocker: map.py is carrying too many responsibilities at once: topbar, progression summary, relic strip, lore text, map graph, node rendering, archon forecast, biome panel, and top navigation buttons. This is the central Patch 0 UI hotspot.
- major: node type normalization and naming are partly duplicated inside map.py with special-case logic for guide/path/relic/shop/boss, which makes node identity fragile.
- major: map top navigation duplicates cross-screen routing in-screen with deck/shop/codex buttons instead of using a canonical top navigation model.
- medium: labels mix lore-facing naming and systems-facing naming: Ritual, Sombra, Prueba, Elite, Mercader, Santuario, Archonte/Arconte inconsistency.
- medium: multiple side panels and title blocks compete visually with the graph itself; the screen is carrying too much informational surface.
- polish: typo/inconsistency risk in node strings and stage naming remains high because text is hardcoded in the screen.

Duplicated or old systems still active
- topbar rendering is centralized in MapTopBar, but map.py still owns several custom header/button strips outside that shared component.
- node lore, node labels, stage titles, branch lanes, and archon profile text are embedded in map.py instead of being fully data-driven.

Legacy UI layers / labels / transitions
- left/right sidebars and center graph produce an older dashboard-like layout rather than a single strong map read.
- deck/shop/codex top buttons behave like legacy global tabs.

Severity
- overall: blocker

3. Combat HUD
Primary files
- game/ui/screens/combat.py
- game/ui/layout/combat_layout.py
- game/ui/components/topbar.py
- game/ui/system/layers.py
- game/ui/system/modals.py

Mapped issues
- blocker: combat.py is overloaded. It owns encounter UI state, dialogue, telemetry, action FSM, tutorial targets, timers, overlays, art manifest access, and render/layout orchestration. Patch 0 should treat it as a consolidation target.
- major: combat screen keeps multiple debug toggles in runtime state: dialog_debug_overlay, qa_debug_overlay, art_debug_overlay. These are clear signs of a still-exposed supervision/debug layer.
- major: two dialogue systems remain active in the same screen: CombatDialogueController and DialogueRouter. Router appears to be the real priority gate, while controller remains as a legacy/simple trigger adapter.
- major: topbar, voices panel, hand area, actions panel, player HUD, harmony core, and card detail are cleaner than before, but the screen still composes too many behavior layers directly in one file.
- medium: combat_layout.py gives a canonical geometry, but combat.py continues to own additional ad hoc layout-derived rect state like end_turn_rect and harmony_seal_rect.
- medium: DEBUG_UI is still hardcoded True in combat.py.

Duplicated or old systems still active
- CombatDialogueController versus DialogueRouter
- ad hoc rect management versus build_combat_layout()
- QA/debug overlays still embedded in runtime combat screen

Legacy UI layers / labels / transitions
- action state labels such as BLOQUEADO, ACCION INVALIDA, FIN DEL RITUAL, EJECUTAR suggest multiple generations of UX states layered together.
- combat start and voice banners still rely on older TypewriterBanner flow alongside newer routed dialogue.

Severity
- overall: blocker

Duplicated or old systems still active
- reward.py and pack_opening.py both present reward selection flows

Legacy UI layers / labels / transitions

Patch 0 Canonical Priorities
1. Unify transitions
- choose one canonical transition screen
- retire or adapt the other as a wrapper only

2. Unify combat dialogue stack
- keep DialogueRouter as canonical trigger arbiter
- demote CombatDialogueController to adapter or remove it
- reduce simultaneous voice surfaces

3. Split combat screen responsibilities
- move debug/QA overlay state out of combat runtime
- isolate action FSM, dialogue, and layout ownership

4. Simplify map screen
- separate graph render, node metadata, top navigation, and side panels
- remove hardcoded naming inconsistencies

5. Normalize reward/shop/pack flow
- define one canonical pre-shop reward progression
- reduce duplicated pack identity logic

6. Remove placeholder routing in presentation/audio contexts
- credits should not alias to victory
- dialogue should not alias to shop
- intro funnel should have one canonical branded path

7. Reduce codex screen repair logic
- move naming/data normalization upstream
- reduce dependence on placeholder portrait fallback chain
